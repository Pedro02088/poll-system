    users {
        bigint id PK
        varchar name
        varchar email UK "unico"
        timestamp email_verified_at "nullable"
        varchar password "hash bcrypt"
        varchar remember_token "nullable"
        timestamps created_updated
    }
    password_reset_tokens {
        varchar email PK
        varchar token "hash do token do e-mail"
        timestamp created_at "expira em 60 min"
    }
    sessions {
        varchar id PK
        bigint user_id FK "nullable"
        varchar ip_address
        text user_agent
        longtext payload
        int last_activity
    }
    polls {
        bigint id PK
        bigint user_id FK "dono"
        varchar title
        text description "nullable"
        boolean is_anonymous "permite voto sem login"
        timestamp expires_at "nullable - bloqueia voto apos a data"
        timestamps created_updated
    }
    options {
        bigint id PK
        bigint poll_id FK
        varchar text
        timestamps created_updated
    }
    votes {
        bigint id PK
        bigint user_id FK "nullable - null quando anonimo"
        varchar voter_token "nullable - identifica visitante"
        bigint poll_id FK
        bigint option_id FK
        timestamps created_updated
    }
    jobs {
        bigint id PK
        varchar queue "fila dos e-mails"
        longtext payload
        smallint attempts
        int available_at
    }

Todas as FKs usam ON DELETE CASCADE. Voto único garantido no banco por duas uniques em votes: unique(user_id, poll_id) para usuário logado e unique(voter_token, poll_id) para visitante anônimo.
Como o MySQL permite múltiplos NULLs numa unique, os dois cenários convivem na mesma tabela. password_reset_tokens e jobs não têm FK: são tabelas de apoio (recuperação de senha e fila de e-mails).
